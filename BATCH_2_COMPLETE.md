## Technical Details

### Status Workflow
The maintenance request lifecycle:
1. **pending** - Initial status when created
2. **open** - Acknowledged by landlord/employee
3. **in_progress** - Work has started
4. **completed** - Work finished
5. **cancelled** - Request cancelled

### Priority System
- **Low** (green 🟢) - Non-urgent issues
- **Medium** (yellow 🟡) - Standard requests
- **High** (red 🔴) - Urgent issues requiring immediate attention

### Work Order Codes
Auto-generated by database trigger:
- Format: `WO-YYYYMMDD-XXXXX`
- Example: `WO-20260213-00001`
- Unique identifier for tracking

### Role-Based Access Control
- **Tenants**: Can create requests, view their own requests
- **Landlords**: Can create requests, view all requests, update status
- **Employees**: Can view requests, update status
- **RLS Policies**: Enforce permissions at database level

## Database Schema Reminder

The `maintenance_requests` table (created in Batch 0) includes:
- `id` - UUID primary key
- `property_id` - FK to properties (nullable)
- `unit_id` - FK to units (nullable)
- `reported_by` - FK to auth.users (required)
- `assigned_to` - FK to auth.users (nullable, for employee assignment)
- `title` - Text (required)
- `description` - Text (required)
- `status` - Enum: pending, open, in_progress, completed, cancelled
- `priority` - Enum: low, medium, high
- `work_order_code` - Auto-generated unique code
- `created_at` / `updated_at` - Timestamps

## Architecture Notes

### Why Shared Screens Between Roles?
CreateMaintenanceRequestScreen and MaintenanceRequestDetailScreen are shared between:
- LandlordTabs
- TenantTabs
- (Could be added to) EmployeeTabs

Benefits:
- **Code reuse** - Same form logic, validation, mutations
- **Consistency** - Same UX for all roles
- **Maintainability** - Single source of truth
- **Role enforcement** - Permissions handled in screen logic

### Status Update UX
Used `Alert.alert()` with action buttons instead of dropdown/modal because:
- Native feel on both iOS and Android
- Less code than custom modal
- Clear, focused decision points
- Standard pattern users recognize

Could be enhanced with:
- Custom modal with larger touch targets
- Status timeline/history view
- Batch status updates
- Bulk actions for multiple requests
